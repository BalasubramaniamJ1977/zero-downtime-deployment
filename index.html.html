The Problem: You Cannot Pause Payments to Deploy

Every engineering team eventually faces a version of this question: how do we ship new code without breaking things in production? For most applications, a brief downtime window is acceptable — annoying, but survivable.

For a high-volume payments platform, it is not. A payment transaction can span eight microservices, two message brokers, a distributed cache, and an external clearing network — and each hop must complete atomically. There is no "retry later." A dropped message means a failed payment. A confused microservice means a corrupted ledger. A split-brain cache means a duplicate transaction.

⚠ The core problem

We needed to deploy new versions of eight interdependent microservices — each connected via Solace queues and Kafka topics, with distributed state in Hazelcast — without interrupting a single in-flight payment, dropping a single message, or creating cross-version transaction contamination.

The naive approaches all fail in interesting ways:

  • ✗
    Rolling update with Kafka: Pod A (old version) and Pod B (new version) consume from the same Kafka topic simultaneously. If the message format changes, one of them crashes. If business logic changes, you get split-processing of a single transaction flow.
  • ✗
    Maintenance window: Drain all queues, take everything down, upgrade, restart. Customers see downtime. Solace topics buffer messages during the window — but the buffer has a limit. For a 24/7 real-time payment rail, this is simply not possible.
  • ✗
    Basic Blue/Green without drain: Flip the load balancer from Blue to Green. All new traffic goes to Green. But what about the 47 transactions still being processed by Blue? They're mid-flight — Kafka offsets uncommitted, Hazelcast maps being written. Kill Blue now and you corrupt data.

We needed something better. Here's what we built.

The Solution: Blue/Green + Canary + Graceful Drain

✓ Our approach

Two permanently live environments — Blue and Green — that swap roles on each deployment. Traffic shifts gradually via canary. The outgoing environment drains its in-flight messages to completion before a single pod is terminated. No transaction is ever split across environments.

The Transaction Chain

Before designing the deployment strategy, we had to fully understand the transaction flow. Every payment travels through this chain — and every microservice in the chain must be on the same version for a given transaction.

Fig 1 — Forward payment transaction flow

The key insight: every transaction has a canonical environment. Once a payment enters Blue's Solace subscriber, it must complete entirely within Blue's service instances. Green must never pick up a message that Blue started processing.

The Blue/Green Environment Model

We maintain two complete, identical environments — Blue and Green — running in the same Kubernetes cluster in separate namespaces. At any point in time, one is Production and one is the Upgrade target.

Fig 2 — Blue/Green environment state machine

The critical rule: the Solace topic TXN.PROD.# is subscribed to by exactly one environment at a time. This is the hard boundary that prevents cross-environment contamination. Kafka consumer groups reinforce this at the message processing layer.

The Canary Ramp: How We Shift Traffic Safely

We don't flip all traffic to Green at once. Traffic moves in graduated stages, with automated gates that must pass before each promotion:

Fig 3 — Canary traffic promotion stages
At Stage 3, half your transaction volume is being processed by the new version in real-time. If anything is wrong, you know before 100% cutover — with half your risk surface. — Principle behind statistical canary analysis (Netflix Kayenta)

The Hardest Part: Draining In-Flight Messages

Promotion to Green is the easy part. The genuinely hard problem is this: at the moment we decide to fully cut over, Blue has messages in various stages of processing — some read from Kafka but not yet committed, some being processed by the Hazelcast Processor, some waiting for Accounting to respond.

We cannot kill Blue pods while this is happening. We cannot let Green pick up Blue's work either. The answer is a three-layer drain protocol.

Fig 4 — Three-layer drain sequence

Each layer sends a drain complete signal. Only when all three signals are green does Kubernetes begin scaling down Blue pods — and even then, it's stepped, never all at once, to preserve PodDisruptionBudget guarantees.

The Spring Boot PreStop Hook

The drain is triggered by Kubernetes' preStop lifecycle hook, which fires before SIGTERM is sent to the application:

# Kubernetes deployment spec — preStop hook
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c",
        "curl -X POST http://localhost:8080/actuator/drain && sleep 30"]
terminationGracePeriodSeconds: 120

# The /actuator/drain endpoint:
# 1. Sets readiness probe → 503 (removed from k8s Service)
# 2. Solace: session.unsubscribe(liveTopicSubscription)
# 3. Kafka: consumer.pause(partitions) → await commitSync()
# 4. Hazelcast: flushWriteBehind() → await completion
# 5. Liveness stays 200 until done (prevents premature SIGKILL)
⚑ Key design decision

The readiness probe returns 503 immediately on drain start — removing the pod from the Service's endpoint list so no new requests arrive — while the liveness probe stays 200 until drain completes, preventing Kubernetes from killing the pod prematurely. This is the difference between a clean drain and a forced shutdown.

Kafka Consumer Group Isolation: The Invisible Safety Net

Consumer group isolation is what makes this entire approach work at the Kafka layer. Each environment has its own consumer group per service:

Postingposting-blueposting-green

When we promote Green, we use Kafka's offset management to align Green's starting position with Blue's committed offsets. Green picks up exactly where Blue left off — no duplicates, no gaps:

🔑 Critical Kafka settings

acks=all + enable.idempotence=true on producers. enable.auto.commit=false + isolation.level=read_committed on consumers. transactional.id per pod on the Processor. This combination gives us exactly-once semantics end-to-end across the deployment transition.

Hazelcast State Continuity: The Simplest Solution

Our initial design involved replicating Hazelcast state from Blue to Green during promotion. It was complex, brittle, and prone to split-brain conditions. We threw it out.

The solution was simpler: both clusters connect to the same database, with write-through map stores. The DB is always the source of truth. Green doesn't need to copy Blue's state — it just loads from the same DB that Blue has been writing to all along.

# hazelcast.yaml — write-through map store
map-store:
  enabled: true
  write-delay-seconds: 0  # write-through, not write-behind
  class-name: PaymentStateMapStore

# Promotion pre-check:
# green.cluster.mapSize() == blue.cluster.mapSize() ± 2%
# If not → block promotion → trigger re-load from DB

TL;DR — The Six Guarantees

  • ✓No lost messages — Kafka exactly-once semantics + manual commit + drain protocol before any pod shutdown
  • ✓No cross-environment transaction bleed — Solace topic isolation + Kafka consumer group isolation enforced at the infrastructure layer
  • ✓No broken in-flight transactions — PreStop drain hook with 120s grace period ensures every started transaction completes in Blue before scale-down
  • ✓No state corruption — Hazelcast write-through to shared DB; Green loads from DB independently, no cross-cluster replication
  • ✓Instant rollback — Blue stays warm standby (replicas=1) for 30 minutes post-promotion; rollback is a topic re-subscribe and offset reset
  • ✓Automated safety — Argo Rollouts analysis blocks promotion if error rate, latency, or Kafka lag degrades; auto-rollback in <30 seconds
